<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| repeated player turns -> `GameState` | Multi-turn inputs can attempt to force premature completion or state confusion. |
| helper response -> player retry | Correct helper content must not be mistaken for a player-provided answer. |
| completed state -> replay reset | Post-completion commands can attempt to mutate finished story state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-10 | Tampering | final-helper retry gate | high | mitigate | Track `final_helper_given` separately and require a later safe player utterance before completion. |
| T-02-11 | Tampering | completion state | medium | mitigate | Lock completed state against further epilogue advancement; allow only replay reset. |
| T-02-12 | Information Disclosure | sidekick and completion prompts | high | mitigate | Preserve no-internal-instruction guardrails and keep prompt-injection behavior active in full suite. |
| T-02-13 | Spoofing | replay command detection | low | mitigate | Gate replay reset on `state.complete is True`; mid-story replay-like text must not erase progress. |
| T-02-14 | Denial of Service | repeated wrong/unrelated inputs | medium | mitigate | Cap hint escalation at final helper and offer concrete choices for unrelated repeats. |
| T-02-SC | Tampering | npm/pip/cargo installs | high | mitigate | No new package installs are planned; if an executor adds one, stop for package legitimacy audit and human verification. |
</threat_model>
